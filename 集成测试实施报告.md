## 一、集成测试需求分析

### 1.1 核心特性识别

通过深入分析 DrawAnywhere 的代码架构，我们识别出以下需要集成测试的关键特性：

#### **特性 1: 手写笔感知绘制 (Stylus-Aware Drawing)**
- **代码位置**: `StylusAwareDrawing.kt`
- **功能描述**: 
  - 检测触控笔按钮状态（主按钮/副按钮/双按钮）
  - 根据按钮状态自动切换笔刷类型（Pen ↔ Eraser）
  - 笔画结束后恢复之前的笔刷类型
- **集成点**: 
  - `stylusAwareDrawing Modifier` → `DrawViewModel.startStroke/updateStroke/finishStroke`
  - `DrawViewModel.resolvePenType()` → `PenType` 切换逻辑
- **为什么需要集成测试**: 
  - 涉及手势事件处理与 ViewModel 状态管理的协同
  - 笔刷类型临时切换和恢复的逻辑复杂
  - 需要验证完整的笔画生命周期

#### **特性 2: 画布穿透模式 (Canvas Passthrough)**
- **代码位置**: `MainService.kt`, `DrawViewModel.kt`
- **功能描述**:
  - 通过 `FLAG_NOT_TOUCHABLE` 允许触摸事件穿透画布
  - 与自动清除画布功能联动
  - 影响第二抽屉的按钮固定状态
- **集成点**:
  - `ViewModel.canvasPassthrough` → `Service.handleCanvasPassthrough()` → `LayoutParams.flags`
  - `autoClearCanvas` → 隐藏画布时清除路径并禁用穿透
- **为什么需要集成测试**:
  - 涉及多个状态字段的联动变化
  - 需要验证状态一致性
  - 与 WindowManager 的集成需要在真实环境测试

#### **特性 3: 工具栏位置管理 (Toolbar Position)**
- **代码位置**: `MainService.kt`, `DrawViewModel.kt`, `PreferencesManager.kt`
- **功能描述**:
  - 支持直接设置位置和偏移更新
  - 位置验证和屏幕边界限制
  - 使用 DataStore 持久化工具栏位置
- **集成点**:
  - `ViewModel.updateToolbarPosition()` → `ServiceState.toolbarPosition`
  - `saveToolbarPosition()` → `PreferencesManager.saveServiceState()` → DataStore
- **为什么需要集成测试**:
  - 涉及异步持久化操作
  - 位置计算逻辑需要验证
  - 保存和加载的完整性测试

#### **特性 4: 撤销重做机制 (Undo/Redo)**
- **代码位置**: `DrawController.kt`
- **功能描述**:
  - 最多支持 50 步操作历史
  - 支持 AddPath、ErasePath、ClearPaths 三种操作
  - 新操作清空 redo 栈
- **集成点**:
  - `ViewModel.undo/redo()` → `Controller.undo/redo()` → StateFlow 更新
  - `canUndo/canRedo` 状态流响应
- **为什么需要集成测试**:
  - 虽然是 Controller 的核心逻辑，但需要通过 ViewModel 暴露
  - 需要验证状态流的正确发射
  - 撤销深度限制的边界情况

#### **特性 5: 状态持久化 (State Persistence)**
- **代码位置**: `PreferencesManager.kt`, `DrawViewModel.kt`
- **功能描述**:
  - 使用 DataStore 保存 UI 状态和服务状态
  - 选择性持久化（只保存重要用户偏好）
  - 启动时自动加载保存的状态
- **集成点**:
  - `ViewModel` 初始化 → `PreferencesManager.getSavedUiState/getSavedServiceState`
  - `ViewModel.uiState` 变化 → 自动触发 `saveUiState`
- **为什么需要集成测试**:
  - 涉及真实的 DataStore I/O 操作
  - 需要验证序列化和反序列化
  - 异步操作的时序控制

### 1.3 集成测试策略

基于上述分析，我们采用以下测试策略：

1. **部分 Mock**: Mock PreferencesManager 以避免真实的磁盘 I/O，但使用真实的 DrawController
2. **TestDispatcher**: 使用 `StandardTestDispatcher` 控制协程执行时序
3. **StateFlow 断言**: 直接访问 `.value` 进行同步断言
4. **场景驱动**: 设计真实用户场景而非孤立的功能测试

| 维度 | 单元测试 (test/) | 集成测试 (androidTest/) |
|-----|-----------------|----------------------|
| **测试对象** | 单个类或方法 | 多个类的协作 |
| **Mock 程度** | 高度 Mock | 部分 Mock |
| **运行速度** | 快（秒级） | 慢（分钟级） |
| **依赖环境** | 无需 Android | 需要设备/模拟器 |
| **测试数量** | 150+ | 46 |
| **覆盖场景** | 边界情况、算法逻辑 | 组件集成、工作流 |
| **典型用例** | DrawControllerTest | DrawViewModelIntegrationTest |

**互补关系**: 
- 单元测试确保每个组件内部逻辑正确
- 集成测试确保组件之间协作正常
- 两者结合提供全面的质量保障
