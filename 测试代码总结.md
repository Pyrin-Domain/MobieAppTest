# DrawAnywhere 测试代码总结文档

## 一、测试概览

DrawAnywhere 项目采用了完善的单元测试策略，涵盖了数据模型、工具函数、控制器、视图模型和偏好管理等多个层面。测试代码位于 `app/src/test/java/com/shezik/drawanywhere/` 目录下，共包含 **7 个测试类**，总计约 **150+ 个测试用例**。

### 测试技术栈

- **JUnit 4**: 基础测试框架
- **MockK**: Kotlin Mock 框架，用于模拟依赖
- **Robolectric**: Android 单元测试框架，提供 Android 环境模拟
- **kotlinx.coroutines.test**: 协程测试支持
- **Android Compose UI**: 用于几何图形和颜色类的测试

### 2.2 DataClassTest（数据类测试）

**文件路径**: `DataClassTest.kt`  
**测试类型**: Robolectric 单元测试  
**测试数量**: 20 个  
**运行环境**: SDK 33 (Android 13)

#### 功能说明
全面测试应用中的核心数据类和枚举类型，确保数据结构的正确性和不可变性。

2. **自定义值设置**: 验证可以正确设置自定义的笔刷参数
3. **copy 方法不可变性**: 测试 Kotlin data class 的 copy 方法是否创建新实例而不修改原对象

##### **PathWrapper 测试**（路径包装器）
4. **创建测试**: 验证 PathWrapper 能够正确存储路径点、颜色、宽度和透明度
5. **缓存路径生成**: 测试 cachedPath 属性是否能够从点列表生成有效的 Path 对象
6. **缓存失效机制**: 验证 invalidatePath() 方法能够标记缓存为无效，下次访问时重新生成
7. **释放路径**: 测试 releasePath() 方法能够正确清理并返回缓存的 Path 对象

##### **ServiceState 测试**（服务状态）
8. **默认状态**: 验证工具栏初始位置为 (32f, 64f)，工具栏激活状态为 true
9. **自定义状态**: 测试可以设置自定义的工具栏位置和激活状态

##### **UiState 测试**（UI 状态）
10. **默认 UI 状态**: 验证画布可见、非穿透模式、自动清除关闭、启动时可见等默认行为
11. **当前笔刷配置获取**: 测试 currentPenConfig 属性能够根据当前笔刷类型返回对应配置
12. **自定义笔刷配置映射**: 验证可以为不同笔刷类型设置独立的配置参数

##### **辅助函数测试**
13. **defaultPenConfigs 函数**: 验证默认笔刷配置映射包含 Pen 和 StrokeEraser 两种类型，且橡皮擦宽度为 50f

##### **DrawAction 测试**（绘制动作）
14. **AddPath 动作**: 测试添加路径动作能够正确封装 PathWrapper
15. **ErasePath 动作**: 测试擦除路径动作的正确性
16. **ClearPaths 动作**: 测试清除所有路径动作

##### **枚举类型测试**
17. **PenType 枚举**: 验证包含 Pen 和 StrokeEraser 两个值
18. **StrokeModifier 枚举**: 验证包含 None、PrimaryButton、SecondaryButton、Both 四个修饰符值

#### 设计亮点
- **不可变性验证**: 通过 copy 方法测试强调 data class 的不可变特性
- **缓存机制测试**: 深入测试 PathWrapper 的懒加载缓存和失效机制
- **完整性覆盖**: 覆盖所有数据类和枚举，确保数据结构稳定可靠

### 2.3 DrawUtilsTest（绘图工具函数测试）

**文件路径**: `DrawUtilsTest.kt`  
**测试类型**: Robolectric 单元测试  
**测试数量**: 15 个  
**运行环境**: SDK 33 (Android 13)

#### 功能说明
测试绘图相关的数学计算和几何工具函数，确保距离计算、中点计算和路径生成等底层算法的准确性。

#### 核心测试内容

##### **距离计算函数**
1. **distanceSquared - 同一点**: 验证相同点的距离平方为 0
2. **distanceSquared - 不同点**: 使用经典的 3-4-5 直角三角形验证 (0,0) 到 (3,4) 的距离平方为 25
3. **distance - 同一点**: 验证相同点的欧几里得距离为 0
4. **distance - 不同点**: 验证 (0,0) 到 (3,4) 的距离为 5（勾股定理）

##### **点到线段距离计算**
5. **点在线段上**: 验证点 (5,0) 到线段 [(0,0)-(10,0)] 的距离为 0
6. **点在线段上方**: 验证点 (5,5) 到水平线段的垂直距离为 5
7. **点在线段起点之前**: 验证点 (-5,0) 到线段的距离使用起点计算，结果为 5
8. **点在线段终点之后**: 验证点 (15,0) 到线段的距离使用终点计算，结果为 5
9. **线段退化为点**: 验证当线段起点和终点相同时，退化为点到点的距离计算

##### **中点计算**
10. **标准中点**: 验证 (0,0) 和 (10,20) 的中点为 (5,10)
11. **相同点中点**: 验证相同点的中点仍为该点本身

##### **路径生成函数 (pointsToPath)**
12. **空列表**: 验证空点列表生成空的 Path 对象
13. **单点**: 验证单点列表能够创建包含 moveTo 操作的 Path
14. **多点平滑路径**: 验证三个点能够生成平滑的曲线路径（使用二次贝塞尔曲线）
15. **两点直线**: 验证两个点生成直线路径

#### 算法特点
- **边界情况处理**: 全面测试点到线段距离的各种边界情况（投影在线段内、外、端点重合等）
- **数值精度**: 使用浮点数比较容差 0.001f，确保计算精度
- **几何正确性**: 基于经典几何定理（勾股定理）验证算法正确性

### 2.5 DrawControllerTest（绘图控制器测试）

**文件路径**: `DrawControllerTest.kt`  
**测试类型**: 纯单元测试  
**测试数量**: 13 个  
**测试策略**: 直接实例化测试

#### 功能说明
测试 DrawController 的核心绘制逻辑，包括路径创建、更新、撤销重做、橡皮擦等功能。这是应用的绘制引擎核心。

#### 核心测试内容

##### **路径基本操作**
1. **创建路径**: 验证 createPath 能够将点添加到 pathList，并创建包含该点的新路径
2. **更新最新路径**: 验证 updateLatestPath 能够将新点追加到当前正在绘制的路径
3. **完成路径**: 验证 finishPath 能够将路径添加到撤销栈，并更新 canUndo/canRedo 状态

##### **撤销重做机制**
4. **撤销操作**: 验证 undo 能够从 pathList 移除最后一条路径，并更新撤销/重做状态
5. **重做操作**: 验证 redo 能够恢复之前撤销的路径
6. **多次撤销**: 测试连续撤销两条路径后，pathList 只剩一条，且仍可继续撤销
7. **撤销深度限制**: 创建 55 条路径，验证最多只能撤销 50 次（maxUndoDepth），防止内存溢出
8. **新操作清空重做栈**: 验证在撤销后执行新操作会清空重做栈（标准撤销重做行为）

##### **清除功能**
9. **清除所有路径**: 验证 clearPaths 能够移除所有路径，并将清除操作加入撤销栈

##### **橡皮擦功能**
10. **笔画橡皮擦**: 验证切换到 StrokeEraser 后，点击已有路径附近能够删除整条路径

##### **边界情况**
11. **未配置笔刷时创建路径**: 验证在没有调用 setPenConfig 的情况下创建路径会抛出 IllegalStateException
12. **空路径不被添加**: 验证 finishPath 时如果路径没有点，不会将其添加到 pathList
13. **canClearPaths 状态**: 验证 canClearPaths 在有路径时为 true，清除后为 false

#### 关键设计模式
- **命令模式**: 通过 AddPath、ErasePath、ClearPaths 等 DrawAction 实现可撤销操作
- **撤销栈深度限制**: 通过 maxUndoDepth (50) 防止无限累积导致内存问题
- **状态流响应式**: 使用 StateFlow 暴露 canUndo、canRedo、canClearPaths 状态

### 2.7 PreferencesManagerRobolectricTest（偏好管理器测试）

#### 功能说明
测试 PreferencesManager 的数据持久化功能，使用 Robolectric 提供真实的 Context 和 DataStore 环境，验证 UI 状态和服务状态的保存与恢复。

#### 测试架构

##### **环境设置**
- 使用 `RuntimeEnvironment.getApplication()` 获取 Robolectric 提供的 Application Context
- 使用 `StandardTestDispatcher` 控制协程执行
- 在 `@After` 中重置 Main dispatcher，避免测试间干扰

#### 核心测试内容

##### **UI 状态持久化**

9. **getEnumValueOrDefault - null 返回默认值**: 验证传入 null 时返回默认枚举值
10. **getEnumValueOrDefault - 无效值返回默认值**: 验证传入不存在的字符串时返回默认值

#### 持久化策略分析

##### **UI 状态持久化字段**
- ✅ **持久化**: autoClearCanvas、visibleOnStart、currentPenType、toolbarOrientation、penConfigs
- ❌ **不持久化**: canvasVisible（从 visibleOnStart 派生）、canvasPassthrough（临时状态）、抽屉状态（UI 临时状态）

##### **服务状态持久化字段**
- ✅ **持久化**: toolbarPosition（用户自定义位置很重要）
- ❌ **不持久化**: positionValidated、toolbarActive（运行时状态）

#### 设计亮点
- **选择性持久化**: 只保存重要的用户偏好，避免保存临时 UI 状态
- **枚举安全解析**: 提供容错机制，无效值回退到默认值
- **DataStore 集成**: 使用现代 Android DataStore API 替代 SharedPreferences
- **Robolectric 真实环境**: 在测试中使用真实的 DataStore 实现，确保与实际运行环境一致

## 四、关键设计模式与最佳实践

### 4.1 MVVM 架构测试模式

```kotlin
// ViewModel 测试标准模式
@Test
fun testFeature() = runBlocking {
    // Given - 准备测试数据
    val input = someTestData
    
    // When - 执行被测方法
    viewModel.someMethod(input)
    testDispatcher.scheduler.advanceUntilIdle()
    
    // Then - 验证状态和副作用
    assertEquals(expected, viewModel.state.value.property)
    verify { mockDependency.calledMethod() }
}
```

### 4.2 撤销重做测试模式

```kotlin
// 标准撤销重做测试流程
@Test
fun testUndoRedo() = runBlocking {
    // 1. 创建操作
    controller.createPath(point)
    controller.finishPath()
    
    // 2. 验证可撤销
    assertTrue(controller.canUndo.value)
    
    // 3. 执行撤销
    controller.undo()
    assertEquals(0, controller.pathList.size)
    
    // 4. 验证可重做
    assertTrue(controller.canRedo.value)
    
    // 5. 执行重做
    controller.redo()
    assertEquals(1, controller.pathList.size)
}
```

### 4.3 缓存机制测试模式

```kotlin
// 缓存有效性测试
@Test
fun testCaching() {
    // 1. 首次访问生成缓存
    val cache1 = wrapper.cachedPath
    
    // 2. 再次访问应返回同一实例
    val cache2 = wrapper.cachedPath
    assertSame(cache1, cache2)
    
    // 3. 修改数据后缓存失效
    wrapper.invalidatePath()
    val cache3 = wrapper.cachedPath
    assertNotSame(cache2, cache3)  // 新实例
}
```

### 4.4 状态流测试最佳实践

    // 测试代码

### 5.3 常见问题与解决方案

#### **问题 1: Robolectric 测试失败**
- **原因**: SDK 版本不匹配或缺少 JVM 参数
- **解决**: 确保 `@Config(sdk = [33])` 与项目目标一致，添加 `-noverify` JVM 参数

#### **问题 2: 协程测试超时**
- **原因**: 未正确推进 TestDispatcher
- **解决**: 使用 `testDispatcher.scheduler.advanceUntilIdle()`

#### **问题 3: MockK 验证失败**
- **原因**: suspend 函数未使用 coVerify
- **解决**: 对协程函数使用 `coVerify { ... }`

#### **问题 4: StateFlow 断言失败**
- **原因**: 异步更新未完成
- **解决**: 使用 `advanceUntilIdle()` 或直接访问 `.value`
