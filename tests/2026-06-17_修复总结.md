## 1. Stroke Controls（笔画粗细/透明度）滑动修复

### 问题描述
`_test_stroke_controls()` 中，笔刷粗细和透明度调节在 tool controls 弹窗的第二个页面（Compose HorizontalPager）。需要打开弹窗后**在弹窗内部从右向左滑动**才能翻到第二页。

### 根因
原代码使用全屏滑动（`actions.swipe("left")`），但弹窗只是一小块区域，全屏滑动手指落在弹窗外面，HorizontalPager 收不到手势。

### 修复
新增 `_swipe_inside_tool_popup()` 方法：
- 解析 UI hierarchy，找到弹窗内的 page-1 元素（Pen、Stroke Eraser、Tools）确定 bounds
- 在弹窗范围内 95%→5% 滑动
- 速度 0.15s（快速翻页）

## 5. Undo/Redo 测试修复

### 问题描述
`_test_undo_redo()` 中，redo 按钮位于 toolbar 第二抽屉（drawer 2）的右侧。需要：
1. 点击 `expand_toolbar` 展开第二抽屉
2. 在 toolbar 区域内**从右向左滑动**来滚动出 redo 按钮

### 根因分析

`_swipe_inside_toolbar()` 原本通过 UI hierarchy 检测 toolbar 元素的 bounds 来决定滑动范围。但它把 drawer-2 的元素（`Collapse toolbar`、`Redo`）也纳入了 bounds 计算：

| 运行 | Toolbar bounds | 滑动起点 | 结果 |
|---|---|---|---|
| 正常 | `[243, 747]`（仅 drawer 1） | `696`（在 toolbar 上） | ✅ redo 找到 |
| 异常 | `[243, 1060]`（含 drawer 2） | `1019`（超出可交互区域） | ❌ 滑动无效 |

**原因**：Compose `horizontalScroll` 要求触摸起点必须在组件范围内。当 `Collapse toolbar`（x=1060）被检测到时，bounds 右边界膨胀，滑动起点落到 toolbar 交互区域之外，滚动不生效。

### 修复方案

`_swipe_inside_toolbar()` 仅使用 **drawer-1 元素** 计算 bounds：

```python
drawer1_patterns = [
    "Expand toolbar", "Undo", "Clear canvas",
    "Tool controls", "Color picker", "Show/Hide canvas"
]
# 不再包含 Collapse toolbar、Redo
```

滑动方向 `"left"` 时：从 drawer-1 右边界（`tx2 - 10`）滑到左边界（`tx1 + 10`），确保触摸始终在可交互区域内。

## 6. Redo 查找超时优化

### 时间线

| 尝试 | timeout 值 | 实际耗时 | 说明 |
|---|---|---|---|
| 初始 | `DEFAULT_TIMEOUT` (15s) | 15188ms | 滑动未生效，等满 15s |
| 第1次 | `DEFAULT_TIMEOUT * 0.1` (1.5s) | 1687ms | 滑动已生效，动画延迟 |
| 第2次 | `DEFAULT_TIMEOUT * 0.02` (0.3s) | 781ms | `_execute_strategy` 下限 0.5s |
| 第3次 | `SHORT_TIMEOUT` (5s) | 5235ms | 冗余等待 |
| **最终** | **`1.0` (1s)** | **~800ms** | **动画延迟可接受** |

### 关键发现

- `locator.py` 的 `_execute_strategy()` 有 **0.5s 下限**（`timeout = max(timeout, 0.5)`），传更小的值无效
- Redo 延迟来自模拟器上 Compose 滚动动画，不是 find 策略的问题
- 最终方案：`time.sleep(ANIMATION_WAIT)` + `timeout=1.0`，在延迟和可靠性间平衡
